Documentation - a Necessary Evil?

Throughout my career as a developer, many things have changed. But the one common thread that I’ve always encountered is a significant number of my fellow developers stating how much they “hate documentation”.
The problem isn’t necessarily the document itself, more the process, raising questions such as:
* Why do we bother, no one reads them anyway?
* Why do we bother, as soon as I start coding, the document is out-of-date?
* Why do we bother, I might as well get straight into coding and work out the problems as I go along?

All of these are good questions, but IMHO, they all miss the main purposes of the process; to communicate and analyse.

[Where I do have a problem with hand-written documentation is when it is merely repeating information from elsewhere.  This is especially true for API documentation and Test documentation.  Both of these should be driven by a comment markup system direct from the code/tests – Doxygen, Sandcastle, etc.]

I’m collabaritively working on a very complex piece of software at the moment.  Before we started coding, we created a 1-2 page summary document, detailing: Background, Stakeholders, Expectations, Timeframe.  This is important, as it not only sets expectations, but also helps us to clarify what we are trying to achieve (to make sure we don’t build the wrong solution).  Once we understood the problem, and the expectations, we started work on the design.  Within a few days we’d sketched out a 20 page (mostly diagrams – using PowerPoint) design summary, showing what the system did.  This was followed by a spreadsheet which nicely captured our decision logic, depending on the state and action.

It only took a few days to get this far, but during that timeframe it has already gone through a number of iterations – and has changed again since we started coding.  By keeping the information simple and freeform, we have been able to easily edit it, and re-analyze the entire system without thinking about syntactic and semantic code issues.  If we were to have gone straight to coding, we would probably be 2-3 weeks behind where we are now – and we would be much more reluctant to make large changes.

Being clear and concise is key to good technical documentaiton.  However, the most important thing, just like with code, is making the document easy to maintain!

Depth First Development

When developing systems (especially new ones), there is often a desire to get started on as much as possible, as soon as possible.  When dealing with many new concepts, or risky items, starting things sooner helps to settle the nerves, and, verify that the system is actually buildable.  During the implementation phase, everything in the system inches forwards to completion.  This incremental, across the board, development is known as Breadth First Development.  However, by developing an entire system using breadth first, it essentially means that the project is never complete, until it is 100% finished across all of the features under development.

Now we can see that whilst the project is not feature complete, it is 100% finished in each feature that has been developed.

This bring a number of enormous benefits, including:

  • You can deliver the fully working, ableit not fully complete, project into Test, Verification and perform Product Demo’s, etc…
  • You have the ability to ship the project, right now – even though it isn’t feature complete, each implemented feature is completely finish.
  • Means that if development does overrun, you can chop features out and reap the rewards.  You can chop features out in breath first, but you have already invested time (and perhaps already have dependencies on some parts of them).
  • Some lower priority features may no longer be required.
  • Some lower priority features may have been replaced with different features.  This is generally termed “Feature Creep”, and whilst many people wish it didn’t exist, it is a natural (and inevitable) thing.  Don’t fight it, learn to deal with it.

It isn’t a new concept (and I’m not suggesting it is my own concept), but it is definitely worth thinking about.  If you do decide to follow it, let me know why, and how you get on.

Science Projects

Back when I was studying science at school, we always followed the same steps when conducting projects:

  • Hypothesis
  • Practical
  • Conclusion

Irrespective of whether the results of the practical aligned with the hypothesis, it was always important to analyse the results in the conclusion. If they aligned, it was important to ensure that sufficient rigour was taken during the practical to make sure that the results were correct. If they did not align, then it was important to understand why, and why the intuition was incorrect.

Since entering the business world, this analytical approach to tasks does not seem to occur very often. Whether it be for a meeting (the steps are the same, but they are called different things: agenda, discussion, minutes), a project, a prototype, or any number of other tasks. I’m not too sure why this is the case, but when these steps are not taken, the results of those tasks generally seems to be of lower quality – or in worse case, they get repeated, over, and over, and over again.
